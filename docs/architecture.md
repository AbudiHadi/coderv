# Architecture — claude-docs-toolkit

> **Generated by `/docify` on 2026-04-24 from commit `fbb1954`.**


## Three parts

```
┌──────────────────────────────────────┐
│  skills/        → what Claude runs    │
│  templates/     → what /docify copies │
│  install.sh     → how it all lands    │
└──────────────────────────────────────┘
```

Everything else (README, VERSION, CHANGELOG, LICENCE) is metadata around these three.

## Skills

Each skill is a folder: `skills/<name>/SKILL.md`.

A `SKILL.md` has:

1. **YAML frontmatter** — `name`, `description`, `user-invocable`, `argument-hint`. The `description` is what Claude Code's skill picker reads. <!-- src: skills/docify/SKILL.md:1-13 -->
2. **TRIGGER / SKIP block inside the description** — natural-language phrases that should surface this skill even without the slash prefix. <!-- src: skills/docify/SKILL.md:5-9 -->
3. **Step-by-step body** — markdown instructions Claude follows when the skill is invoked.

Shape is consistent across all 5 skills: discover → plan → act → report.

## Templates

`templates/*.md` — fresh-project scaffolds. Copied (not touched) by `/docify` when it sets up a new project:

| Template | What it scaffolds |
|---|---|
| `templates/CLAUDE.md` | Full rule set with `<!-- coderlap:rule:* -->` markers |
| `templates/DECISIONS.md` | Empty ADR log + template block |
| `templates/KNOWN-ISSUES.md` | Empty recurring-bug log + template block |
| `templates/SESSIONS.md` | Empty session-handoff log + template block |

Edits to these templates propagate to **new projects only** — existing projects keep what they have. `/docify` with rule-marker merging can add missing *rules* without overwriting user content.

## Installer flow

```
./install.sh              → copy skills/ → ~/.claude/skills/
./install.sh --project    → copy skills/ → ./.claude/skills/
./install.sh --force      → overwrite existing installations
./install.sh --uninstall  → remove only skills with the toolkit marker
```

<!-- src: install.sh -->

The `--uninstall` safety net: after copy, every `SKILL.md` gets this trailing line:

```html
<!-- claude-docs-toolkit -->
```

`--uninstall` greps for that marker before deleting — so a user with unrelated skills named `before` (etc.) won't lose them.

## The web installer

`curl -fsSL https://coderv.dev/install.sh | bash` hits a separate wrapper hosted on the website. That wrapper clones this repo to a tmp dir and runs the real `install.sh` from inside. One-liner UX without giving users a raw bash payload to trust blindly — they can read the wrapper at <https://coderv.dev/install.sh> and the actual installer in this repo.

## How changes propagate to users

1. Edit a `SKILL.md` here.
2. Bump `VERSION` + add `CHANGELOG` entry.
3. Commit, tag, push, create GitHub release.
4. User re-runs `curl -fsSL https://coderv.dev/install.sh | bash -s -- --force` or `cd coderv && git pull && ./install.sh --force`.

There is no auto-update — this is intentional. Skills shouldn't change silently under users.

## Relationship to the website repo

The website (`coderv-docs`) is **separate**. It:
- Renders marketing / docs pages.
- Hosts the web installer wrapper.
- Has its own version number that must be kept in sync (both bumped on every release).
- Duplicates the LICENCE (must stay identical).
