## /docify

**File:** `skills/docify/SKILL.md` <!-- src: skills/docify/SKILL.md -->
**One-line:** Generate a professional `docs/` folder from an existing codebase.
**When to use:** once per project, or `--refresh <file>` to regenerate one doc.

### TRIGGER (suggest even without `/docify` prefix)

Bare words: `docs`, `doc`, `documentation`, `readme`, `README`.

Phrases: "write docs", "create docs", "generate docs", "make docs", "make a doc", "make a full docs", "need docs", "no docs exist", "this project has no README", "document this project", "explain the codebase", "document what we have", "I need docs", "our docs are outdated", "fresh docs", "bootstrap docs", "docs are missing", "docs outdated". <!-- src: skills/docify/SKILL.md:5 -->

### SKIP

When the user wants to edit ONE specific existing doc file by hand ("fix a typo in api.md"), OR when "docs" clearly means something unrelated ("the docs page on our website", medical doc, etc.). On ambiguous short mentions ("docs?"), the skill asks for clarification first. <!-- src: skills/docify/SKILL.md:7-9 -->

### Safety rules

1. Never hallucinate — every claim cites `<!-- src: path:line-range -->`.
2. Never overwrite existing docs without approval.
3. CLAUDE.md merge preserves existing user rules.
4. Plan-first — user approves before file writes.
5. Drafts-first — writes `.draft.md` files; user promotes via `/docify approve <name>`.

## /before

**File:** `skills/before/SKILL.md` <!-- src: skills/before/SKILL.md -->
**One-line:** Pre-code checklist — reads docs, plans, waits for approval.
**When to use:** before any non-trivial code change.

### TRIGGER

Phrases: "add X / build X / implement X / refactor / rewrite / rename / integrate / migrate / port / extract / set up / wire up". Also "take your time", "think about this", or first touch of a module this session. <!-- src: skills/before/SKILL.md -->

### SKIP

"quick fix / just / small change / tiny edit / typo", session-local undos, documentation-only edits.

### What it does

Reads CLAUDE.md + relevant docs, greps repo for prior art, checks DECISIONS.md for applicable ADRs, checks KNOWN-ISSUES.md for recurring bugs, states a plan with files it expects to touch, **waits for approval**, then codes.

## /decision

**File:** `skills/decision/SKILL.md` <!-- src: skills/decision/SKILL.md -->
**One-line:** Log an ADR in 30 seconds while the choice is fresh.
**When to use:** right after picking a library / pattern / trade-off.

### TRIGGER

User picks between options ("X or Y?", "should we use A vs B"), asks "why did we pick X?", mentions "architectural decision", "ADR", "tech choice", or accepts a trade-off. <!-- src: skills/decision/SKILL.md -->

### SKIP

Trivial picks with no trade-off (variable names, etc.).

### What it does

Finds the highest ADR number, walks user through 5 questions (Context, Decision, Alternatives, Consequences, Decider), prepends a new ADR to `docs/DECISIONS.md`. Supersession is explicit — old ADR gets updated status, new ADR references it.

## /ship

**File:** `skills/ship/SKILL.md` <!-- src: skills/ship/SKILL.md -->
**One-line:** Pre-commit checklist — auto-updates docs + validates citations + drafts commit.
**When to use:** before every commit that touches multiple files.

### TRIGGER

"commit / push / ready to ship / I'm done / finalize / let's commit / commit and push / ready to merge / draft commit message / write commit message". <!-- src: skills/ship/SKILL.md -->

### SKIP

When user is only asking about git status without intent to commit.

### What it does

Reads `git diff`, categorises changes (routes / components / schema / etc.), offers to update the relevant doc itself (not just ask), re-validates every `<!-- src: -->` citation, drafts a why-focused commit message, suggests follow-ups (`/decision` if design choice, KNOWN-ISSUES entry if non-obvious bug).

## /session

**File:** `skills/session/SKILL.md` <!-- src: skills/session/SKILL.md -->
**One-line:** End-of-session handoff so the next session picks up cleanly.
**When to use:** end of day, `/session last` at the start of next session.

### TRIGGER (end)

"I'm done for today / wrap up / see you tomorrow / stopping here / pausing / EOD / taking a break / lunch break". <!-- src: skills/session/SKILL.md -->

### TRIGGER (start — `/session last`)

"where did I leave off? / what was I doing? / pick up where I left off / what's next? / status? / continue from yesterday". <!-- src: skills/session/SKILL.md -->

### SKIP

In the middle of active work with no break in sight.

### What it does

Pulls recent `git log`, asks user to fill in 4 fields (what shipped / in flight / gotchas / next), prepends entry to `docs/SESSIONS.md`.

## Adding a new skill

2. Include both `TRIGGER` and `SKIP` blocks in the description.
3. Body: steps Claude should follow, in order, with examples.
4. Test via `./install.sh --force` → start Claude Code → verify skill picker sees it.
5. Add to CHANGELOG as a new minor version.
6. Update this file (`/docify --refresh skills`).
