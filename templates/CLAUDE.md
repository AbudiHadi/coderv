# Project AI Instructions

> This file tells Claude how to work in this project. It's loaded automatically every session.
>
> Rules added by the CoderLap toolkit have `<!-- coderlap:rule:* -->` markers so `/docify` can safely re-add missing ones without touching your custom rules.

<!-- coderlap:rule:loop -->
## The loop (every task)

1. Start of session: **`/session last`** — read the last handoff.
2. Before coding: **`/before <task>`** — Claude reads docs, plans, waits for approval.
3. When you make a design choice: **`/decision <title>`** — log it while it's fresh.
4. Before committing: **`/ship`** — auto-updates docs, validates citations, drafts commit.
5. End of session: **`/session`** — write a handoff for next time.

## The three living files in `docs/`

| File | What it holds |
|---|---|
| `docs/DECISIONS.md` | ADRs — why we chose X over Y. Newest at top. Never delete. |
| `docs/KNOWN-ISSUES.md` | Recurring bugs: symptom, root cause, fix, prevention rule. |
| `docs/SESSIONS.md` | End-of-session handoffs. Newest at top. |
| `docs/<AREA>-GAPS.md` (optional) | Missing features per domain. Closed gaps become `shipped YYYY-MM-DD (commit: <hash>)`. |

Plus generated by `/docify`:
- `docs/overview.md`, `architecture.md`, `api.md`, `components.md`, `database.md`, `integrations.md`

<!-- coderlap:rule:principles -->
## Code principles

- **SR** (Single Responsibility) — one thing per function / component.
- **DRY** — shared services, not copy-paste.
- **CC** (Component Composition) — small composable pieces.
- **SoC** (Separation of Concerns) — routes → services → DB.

<!-- coderlap:rule:never-unrequested -->
## Never add unrequested scope

Do not add features, helpers, or refactors the user didn't ask for — even if you see something you'd do differently. Stay inside the task. Mention ideas in your final output; don't implement them.

<!-- coderlap:rule:never-delete-history -->
## Never delete history

- Closed gaps become `shipped YYYY-MM-DD (commit: <hash>)`. Keep the section.
- Superseded ADRs keep their section with `superseded by ADR-NNN` in status.
- Archived docs get a banner, not a deletion.

<!-- coderlap:rule:absolute-dates -->
## Absolute dates only

Write `2026-05-01`, never "next Thursday". Records outlive the session that wrote them.

<!-- coderlap:rule:suggest-followups -->
## Suggest the right skill at the right moment

As you work, watch for moments that deserve a toolkit command and **suggest** it inline (never auto-run). The user may not remember which slash command maps to their intent — you do.

### Listen for these phrases and suggest the matching skill

**`/docify`** — user says: "write docs", "write documentation", "create docs", "generate docs", "no docs exist", "document this project", "explain the codebase", "make docs for X", "I need docs", "our docs are outdated", "fresh docs", "bootstrap docs".

→ Reply: *"I can generate real docs from your code using `/docify` — it scans the project, writes CLAUDE.md + architecture + api + components + more, every claim cited back to a source file. Want me to run it?"*

**`/before`** — user asks to: "add X", "build X", "implement X", "refactor", "rewrite", "rename", "integrate", "migrate", "port", "extract", "set up", "wire up" — and the task is NOT a one-line fix or typo.

→ Reply: *"Before I start, let me run `/before` — it'll read the relevant docs, check for prior art and past decisions, and state a plan for your approval. Takes 10 seconds and catches 80% of misalignment."*

**`/decision`** — user is picking between options, asks "why did we pick X?", mentions a trade-off, accepts a downside in exchange for an upside.

→ Reply: *"This is a real trade-off. Worth logging with `/decision`? 30 seconds now saves re-debating in 6 months."*

**`/ship`** — user says: "commit", "push", "ready to ship", "I'm done", "finalize", "let's commit", "draft commit message".

→ Reply: *"Before committing, let me run `/ship` — it'll check which docs need updating, validate citations are still accurate, and draft a why-focused commit message."*

**`/session`** (end) — user says: "done for today", "wrap up", "see you tomorrow", "stopping here", "pausing", "EOD", "taking a break".

→ Reply: *"Want me to write a `/session` handoff note before you go? Tomorrow-you will thank you."*

**`/session last`** (start) — user says: "where did I leave off?", "what was I doing?", "continue", "remind me", "status".

→ Reply: *"Let me run `/session last` first — it'll tell you exactly where we stopped."*

### Rules

- **Always suggest, never auto-run.** User approves or ignores.
- **One suggestion per moment.** Don't spam the user with multiple skill pitches in one turn.
- **If the user already ran the skill this session, don't re-suggest.** Move on.
- **If the user says "skip it" or "not needed" to a suggestion, respect it.** Don't pitch the same skill again in the same session.

<!-- coderlap:rule:prevention-rules -->
## Bug fixes need prevention rules

When a non-obvious bug is fixed (>15 min to root-cause, or a cause that could easily recur), `/ship` adds an entry to `docs/KNOWN-ISSUES.md` with:

- Symptom (in user's words)
- Root cause (1–3 sentences, the actual mechanism)
- Fix (file paths + line ranges)
- **Prevention rule** — the test/type/lint/checklist item that would catch it next time

If you can't name a prevention rule, the debugging isn't finished.

<!-- coderlap:rule:validate-citations -->
## Citations in docs must stay honest

Generated docs cite source code via `<!-- src: path:line-range -->`. `/ship` re-validates every citation in `docs/**.md` on commit — if the file is gone or the line range is out of bounds, flags it for refresh via `/docify --refresh <file>`.

<!-- coderlap:rule:claude-md-stable -->
## CLAUDE.md stays stable

This file is edited deliberately, not automatically. `/before` reads it every task but never writes to it. `/docify` can add **missing** rules (marked with `<!-- coderlap:rule:* -->`) but never touches existing rules — yours or ours.

Rules without markers are yours. `/docify` leaves them alone forever.

<!-- coderlap:rule:commit-why -->
## Commit messages explain why

Not just what. `/ship` drafts a message in this shape:

```
<type>: <summary in imperative mood>

Why: <1-2 sentences — business or technical reason>
What: <terse list if multi-file>

Closes: <issue ref if any>
```

No AI attribution / Co-Authored-By / emoji unless the project already uses them.

<!-- coderlap:rule:validate-boundaries -->
## Validate at boundaries, trust inside

At project boundaries (user input, external APIs, env vars) validate explicitly. Inside the project, trust your own types and invariants — don't defensively re-validate data that's already been checked.

## Never

- Skip `/before` on a task with multiple files or a new module.
- Commit without running `/ship`.
- Delete an ADR, KI entry, or gap section — update its status instead.
- Write code when the user said "plan first".
- Auto-run a skill. Always suggest, wait for user.

## Always

- Read the docs that `/before` surfaces before coding.
- Let `/ship` update docs for you where it can.
- Log design choices as ADRs via `/decision` immediately, while fresh.
- Write a `/session` note if you leave work in flight.
